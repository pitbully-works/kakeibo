# かけいぼ ― 夢に近づく家計簿

「面倒で続かなかった人でも続く家計簿」を目指した、ビルド不要の自己完結アプリです。
開いた瞬間に「今つかえるお金」がひと目でわかり、記録は 1〜2 タップ。将来は「資産形成総合ライフプラン」へ自然につながります。

- **ひと目でわかるホーム**：主役は「今月あと使えるお金」。
- **1〜2タップで記録**：手入力、またはレシート撮影（任意のOCRで金額を自動読み取り、いつでも手直し可）。
- **端末内だけに保存**：データは `localStorage` に保存し、外部に送信しません。
- **PWA**：ホーム画面に追加でき、2回目以降はオフラインでも起動します。

## 計算仕様（唯一の正）

計算は **`core.js` だけ** に置いてあります。ホーム・まとめ・連携JSONは、
すべて `core.js` の `computeMonth()` の結果を読むだけで、画面ごとに足し算をしません。

### 正式な計算式

```
使える額 = 通常収入 + 臨時収入 － 支出 － 先取り貯金 － NISA積立
```

ホームの「今月 あと つかえるお金」、まとめの「のこり」、連携JSONの
`available_to_spend` は、**必ず同じ値**になります。

### ただひとつの原則：入力口はひとつだけ

お金の出入りは、**すべて「記録」から**入れます。設定には持ちません。

- 収入 … 💴 通常給与／✨ 臨時・賞与／🎁 贈与／🐷 その他臨時
- 支出 … 固定費（🏠 家賃・💡 電気・🔥 ガス・🚰 水道・📱 通信・🔁 サブスク・🛟 保険・📦 その他固定費）も、
  変動費（🥕 食費・🍜 外食・🧴 日用品・🚃 交通・🎨 趣味・🏥 医療・🎁 交際・🐷 その他）も、同じ「記録」

**同じ金額を2か所に書ける作りにしない。だから二重計上が起きません。**
予定額と実績を突き合わせる仕組みも、置き換えのルールも要りません。

固定費と変動費の区別は、まとめ画面の内わけ表示のためだけにあります。
どちらも同じように支出として引かれます。

### 設定に残すもの

「まだ出ていないお金」だけです。

- 先取り貯金（月・予定額）
- NISA積立（月・予定額）
- 夢・目標

どちらも入金確認をしていないため、**予定額**として扱います。
ホームの表記も「先取り貯金・NISA積立の予定額を除いています」で統一しています。

### 給与を記録していない月

収入0として扱い、ホームは「使えるお金」ではなく「今月つかった金額」を表示して、
給与の記録をうながします（マイナスの大きな数字を出さないため）。

```json
{
  "schema_version": "2.1",
  "country_code": "JP",
  "base_currency": "JPY",
  "year_month": "2026-07",

  "income_regular": 290000,
  "income_regular_basis": "actual",
  "income_regular_recorded": true,
  "income_extra": 50000,
  "income_actual_total": 340000,
  "income_net": 340000,

  "fixed_cost": 98000,
  "fixed_cost_items": [
    { "key": "rent", "name": "家賃・住居", "amount": 60000 },
    { "key": "power", "name": "電気", "amount": 12000 }
  ],
  "variable_spend": 20000,
  "spend_total": 118000,

  "planned_set_aside": 73000,
  "accounts": [
    { "type": "CASH_SAVINGS",    "local": "貯金", "basis": "planned", "planned_contribution": 40000 },
    { "type": "TAX_FREE_INVEST", "local": "NISA", "basis": "planned", "planned_contribution": 33000 }
  ],

  "available_to_spend": 149000
}
```

**2.0 からの変更点（ライフプラン側の読み取りを直す必要があります）**

| 旧 (2.0) | 新 (2.1) | 理由 |
| --- | --- | --- |
| `income_net` は設定の手取りのみ | `income_regular` / `income_extra` / `income_actual_total` に分離。`income_net` は実収入合計。すべて記録した実績 | 臨時収入が抜けていた／設定と記録の二重入力をやめた |
| `fixed_cost` は設定値のみ | `fixed_cost` は記録した固定費の合計。`fixed_cost_items` に項目別の実績 | 設定と記録の二重入力をやめた |
| `accounts[].contribution` | `accounts[].planned_contribution` ＋ `basis: "planned"` | 実績と誤解される名前だった |

`type`（汎用の資産クラス）は既存ライフプランアプリと共通です。

## レシートの読み取り

**撮る → 枠で囲む → 読み取る** の3ステップです。撮っただけでは読み取りません。

1. 「📸 レシートを撮る」で、合計の行に近づけて撮影
2. 白い枠を動かして、合計の金額だけを囲む
3. 「🔍 この枠の金額を読み取る」を押す

### 枠は3種類を自動で試す

利用者が合わせた枠から、**そのまま／上下左右5%広げ／上下左右10%狭め** の3つを自動生成します。
枠が少しずれていても拾えるようにするためで、いずれも画像の外にはみ出しません。

### 画像処理の候補

グレースケール＋弱いコントラスト補正／シャープ化／大津の二値化／適応的二値化。

反転の判断は `core.js` の `prepareForOcr()` の中**1か所だけ**で行います。
背景が暗い（白抜き文字）ときに1回だけ反転し、Tesseractへ渡す画像は常に
「明るい背景・暗い文字」になります。無条件に反転する候補は持ちません（二重反転の防止）。

### 段階式（最大9回、取れたら即終了）

| 段階 | 内容 | 回数 |
| --- | --- | --- |
| 1 | 通常枠 × ゆるい補正・大津 | 2 |
| 2 | 広げ枠・狭め枠 × 大津、通常枠 × 適応的 | 3 |
| 3 | シャープ化・適応的・反転など | 4 |

打ち切る条件は **「異なる読み取り実行で同じ金額が2回以上出たこと」だけ** です。
Tesseractは誤読にも高い信頼度を付けるため、1件だけの結果では——信頼度がいくら高くても——打ち切りません。
切り抜き画像は組み合わせごとに1回だけ作って使い回します。

### 候補の採点

多数決だけでなく、次を合わせて0〜100点を付けます。

- 読み取り信頼度（最大40点）
- 複数の枠・処理で同じ金額が出た回数（最大30点）
- カンマの位置が自然か（最大15点／壊れていれば0点）
- OCRが返した文字列の中で中央寄りに現れたか（最大10点）
  ※ 画像上の座標ではありません。bbox を使っていないため画像内の位置は分かりません。行の中ほど・行内の中ほどに現れた数字をわずかに優遇する弱い手がかりです
- 1円以上999,999円以下（範囲外は0点）

**最高点が60点以上で、2位と12点以上の差**があれば自動入力します。
それ以外は自動確定せず、上位3件をタップできる候補ボタンとして出します。
候補をタップしても金額が入るだけで、保存はされません。従来どおり「記録する」で確定します。
候補が1件も無ければ、これまでどおり手入力できます。

### 解像度

画面表示・保存は縮小版、読み取りは長辺3500pxの高解像度版をメモリ内だけに持ちます。
枠は0〜1の比率なので、どちらの画像でも同じ範囲が切り出せます。
高解像度版は保存しません。**読み取りが終わっても解放しません**——枠を直して2回目を読み取るとき、
表示用(1600px)に落ちてしまうためです。解放するのは次の4つの場面だけです。

1. 記録シートを閉じた時
2. 写真を削除した時
3. 新しい写真に置き換えた時（新しい版を入れる前に古い版を解放）
4. **記録を確定した時**＝`localStorage` への保存が成功した直後

4つ目は「保存を試みた時」ではありません。保存に失敗すると記録シートは開いたままになるため、
そこで手放すと、枠を直して読み直したときに表示用画像へ落ちてしまいます。
そのため解放は次の2か所、いずれも `save()` が成功した**直後**に置いています。

- 最初の保存が成功した時
- 容量超過で写真を外し、再保存が成功した時

両方とも失敗して記録を巻き戻す場合は、高解像度版を維持したままシートを残します。

読み取り中に作る加工画像は `readCrop()` のローカル変数なので、関数を抜ければ参照が消えます。

## 更新の届き方（Service Worker）

`sw.js` は用途で2つに分けています。

- **画面（index.html・ナビゲーション）＝ ネットワーク優先**
  キャッシュを先に返すと、`index.html` や `core.js` だけを更新したときに
  古い画面が端末に残り続けます。画面は必ず最新を取りに行き、
  **通信に失敗したときだけ**キャッシュした `index.html` を返します。
- **部品（core.js・アイコン・manifest）＝ キャッシュ優先**
  表示を速くするため。ただし裏でネットワークからも取り直し、次回に備えます。

オフラインでの起動はどちらの場合も維持されます。

キャッシュ名（`kakeibo-v4`）を変えると、`activate` のときに古いキャッシュを丸ごと捨てます。
**アプリを更新したら、この版数を必ず上げてください。**

## 保存について（端末内・容量の話）

データは `localStorage` に保存します。**保存できるのは端末あたりおよそ5MB**です。
スマホの写真は1枚で数MBあるため、そのまま持つと1〜2枚で上限に達し、
保存が失敗して「記録したのに残らない」状態になります。そのため：

- 撮影した写真は、画面表示用に長辺 **1600px** まで縮めます
- 読み取り用には長辺 **3500px** の高解像度版を**メモリ内だけ**に別途持ちます（保存しません）
- 保存するときは、さらに長辺 **900px** / JPEG品質0.6 まで縮めます（1枚おおよそ100KB前後）
- それでも入りきらない場合は、**写真だけを諦めて記録は必ず残し**、その旨を画面に出します
- どうしても保存できないときは、変更を元に戻したうえで理由を表示します（黙って失敗しません）
- せってい画面に使用量を表示し、「写真をすべて消す（記録は残ります）」で空きを作れます

## テストが効いているかの確認（自作の mutation test）

テストを増やしても、それが本当に不具合を捕まえられるとは限りません。
そこで **ソースをわざと壊して、テストが落ちるかどうか** を確かめる簡易な仕組みを自作しました。

```bash
node run-mutations.js
```

**これは mutation testing 全体を網羅するものではありません。**
Stryker のような本格的なツールのように、あらゆる演算子・条件・戻り値を機械的に変異させる
わけではなく、**壊れると困る重要な21パターンを手作業で選んで列挙したもの**です。
選んだ範囲の外に穴が残っている可能性はあります。

対象にしているのは、拡大枠の無効化、はみ出し防止の削除、反転の二重適用、
読み取り後に高解像度画像を捨てる、保存の成否が出る前に解放する、保存の失敗を握りつぶす、
キャッシュの版数を上げ忘れる、古いキャッシュを消さない、といった、
この家計簿で実際に起きた／起きうる壊れ方 **25パターン** です。

結果は `MUTATION-REPORT.md` に書き出されます。1件でも検出できなければ不合格として終了します。

合否の判定には `node --test` の**終了コード**を使います。
出力の表示形式（tap / spec）は Node のバージョンや実行環境で変わるため、
文字列を読み取って判定すると環境によって壊れます。

実際にこの仕組みで2件の穴が見つかり、テストを追加して塞ぎました
（単独結果での早期終了と、保存失敗時の巻き戻し）。

## テストの実行

追加インストールは不要です（Node 18以降の標準機能だけを使います）。

```bash
node --test
```

### GitHub Actions（push のたびに自動実行）

`.github/workflows/test.yml` により、`main` への push と pull request で自動実行されます。
リポジトリの **Actions** タブで結果（緑／赤）を確認できます。

実行される内容は次の3つで、**どれか1つでも失敗すれば赤**になります。

1. `node --test` … 通常テスト
2. `node run-mutations.js` … 自作 mutation test（見逃しが1件でもあれば失敗）
3. `git diff --exit-code` … mutation test がソースを元に戻せているかの確認

外部APIやシークレットは使いません。`package.json` が無いため `npm test` ではなく、
Node 標準のテストランナーを直接呼んでいます。
結果の件数と mutation test のレポートは、実行画面のサマリーに表示されます。
